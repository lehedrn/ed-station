## 阶段 3B 补充：前端文件结构规范 ⭐⭐⭐

> **基于政务服务事项管理系统的经验教训**（2026-04-02）

### ⚠️ 重要：BladeX 框架不支持 `index.vue` 嵌套结构

**问题描述**：
- 访问 `http://localhost:3888/affair/manage` 时页面显示空白
- 浏览器控制台无报错，但页面内容未渲染

**原因分析**：
- BladeX 框架的 `avue-router.js` 中 `formatPath` 函数通过以下逻辑拼接组件路径：
  ```javascript
  ele.component = 'views' + ele[propsDefault.path];
  ```
- 如果后端菜单配置的 `path` 为 `/affair/manage`，则期望的组件路径为：
  ```
  views/affair/manage.vue
  ```
- 但实际文件结构是：
  ```
  views/affair/manage/index.vue  （嵌套结构）
  ```
- `import.meta.glob` 无法正确匹配 `index.vue` 结构

**对比参考**：
```
✅ innermessage 模块（正常）：
src/views/desk/innermessage.vue

❌ affair 模块（问题）：
src/views/affair/manage/index.vue
```

**解决方案**：
1. 将 `views/affair/manage/index.vue` 重命名为 `views/affair/manage.vue`
2. 将子组件 `views/affair/manage/components/MaterialDialog.vue` 移到同级目录
3. 修改组件导入路径：
   ```javascript
   // 修改前
   import MaterialDialog from './components/MaterialDialog.vue';
   
   // 修改后
   import MaterialDialog from './MaterialDialog.vue';
   ```

**修复后的文件结构**：
```
src/views/affair/
├── manage.vue           (主页面)
└── MaterialDialog.vue   (子组件)
```

**经验教训**：
> ⚠️ **BladeX 框架的前端页面必须采用扁平结构**，不能使用 `index.vue` 嵌套结构。
> 参考已正常工作的模块（如 `innermessage`），保持文件结构一致。

### 前端文件结构规范

1. **页面文件结构**：
   ```
   ✅ 推荐：src/views/module/page.vue
   ❌ 避免：src/views/module/page/index.vue
   ```

2. **子组件位置**：
   ```
   ✅ 推荐：src/views/module/Component.vue (与主页面同级)
   ❌ 避免：src/views/module/page/components/Component.vue
   ```

3. **组件导入路径**：
   ```javascript
   // 同级目录
   import ChildComponent from './ChildComponent.vue';
   
   // 父目录
   import ParentComponent from '../ParentComponent.vue';
   ```

## 阶段 5A 补充：后端开发规范 ⭐⭐⭐

> **基于政务服务事项管理系统的经验教训**（2026-04-02 重构）

### ⚠️ 重要：优先使用 MyBatis-Plus，避免原生 MyBatis XML

**初始实现（错误）**：
```java
// AffairMapper.java - 自定义方法
IPage<AffairVO> selectAffairPage(IPage page, AffairVO affairVO);
```

```xml
<!-- AffairMapper.xml - 自定义 SQL -->
<select id="selectAffairPage" resultMap="affairVOResultMap">
    SELECT a.*, dict.type_value AS affairTypeDict
    FROM blade_affair a
    LEFT JOIN blade_dict dict ON a.affair_type = dict.dict_key
    WHERE a.is_deleted = 0
    ...
</select>
```

**正确做法（参考 innermessage 模块）**：

1. **Mapper 只继承 BaseMapper** - 无需自定义方法
2. **使用 Wrapper 类进行字典转换** - 在 Service 层使用 `DictCache.getValue()`
3. **避免 XML 自定义 SQL** - 除非复杂多表关联查询

**BladeX 标准开发模式**：

```
Entity (实体类) → 继承 BaseEntity
       ↓
VO (视图对象) → 包含字典字段（如 affairTypeDict）
       ↓
Mapper → 继承 BaseMapper<Entity>，无自定义方法
       ↓
Service → 继承 BaseServiceImpl<Mapper, Entity>
       ↓
Wrapper → 继承 BaseEntityWrapper<Entity, VO>
       ↓
Controller → 使用 Wrapper.build().pageVO() 转换结果
```

**DictCache 使用**：

```java
// 1. 在 DictEnum 中添加字典枚举
public enum DictEnum {
    AFFAIR_TYPE("affair_type"),
    AFFAIR_STATUS("affair_status"),
    MATERIAL_TYPE("material_type");
}

// 2. 在 Wrapper 中转换字典值
public class AffairWrapper extends BaseEntityWrapper<Affair, AffairVO> {
    @Override
    public AffairVO entityVO(Affair affair) {
        AffairVO affairVO = BeanUtil.copyProperties(affair, AffairVO.class);
        affairVO.setAffairTypeDict(
            DictCache.getValue(DictEnum.AFFAIR_TYPE, affairVO.getAffairType())
        );
        affairVO.setStatusDict(
            DictCache.getValue(DictEnum.AFFAIR_STATUS, affairVO.getStatus())
        );
        return affairVO;
    }
}
```

**Controller 使用 Wrapper**：

```java
@GetMapping("/list")
public R<IPage<AffairVO>> list(Map<String, Object> params, Query query) {
    // 使用 MyBatis-Plus 标准分页
    IPage<Affair> affairPage = affairService.page(
        Condition.getPage(query), 
        Condition.getQueryWrapper(params, Affair.class)
    );
    // 使用 Wrapper 转换字典值
    return R.data(AffairWrapper.build().pageVO(affairPage));
}
```

**经验教训**：
> ⚠️ **后端开发应优先使用 MyBatis-Plus，避免原生 MyBatis XML**。
> 字典值转换使用 `DictCache.getValue()`，性能更优且符合 BladeX 规范。
> 参考已正常工作的模块（如 `innermessage`），保持开发模式一致。

### 3.2 系统字典 SQL 脚本

**文件**: `sql/02_dict/dict_inner_message.sql`

> **注意**: BladeX 字典表结构说明（阶段 3 执行）
> - `code`: 字典编码（如 'message_type'）
> - `dict_key`: 字典键值（**顶级节点使用 -1**）
> - `dict_value`: 字典显示名称
> - `parent_id`: 父主键（顶级节点为 0，子节点指向父级 id）
> - 无 `tenant_id`、`create_time` 字段

```sql
-- =============================================
-- 字典：消息类型
-- 字典编码：message_type
-- 所属模块：内部消息
-- =============================================

-- 1. 插入字典类型（父级，dict_key = -1）
INSERT INTO blade_dict (id, parent_id, code, dict_key, dict_value, sort, remark, is_sealed, status, is_deleted) 
VALUES (100001, 0, 'message_type', '-1', '消息类型', 1, NULL, 0, 1, 0);

-- 2. 插入字典值（子级，dict_key 为实际业务值）
INSERT INTO blade_dict (id, parent_id, code, dict_key, dict_value, sort, remark, is_sealed, status, is_deleted) 
VALUES 
(100002, 100001, 'message_type', '1', '通知', 1, NULL, 0, 1, 0),
(100003, 100001, 'message_type', '2', '公告', 2, NULL, 0, 1, 0),
(100004, 100001, 'message_type', '3', '待办', 3, NULL, 0, 1, 0);
```

**字段说明**:
| 字段名 | 说明 | 示例 |
|--------|------|------|
| `id` | 主键（雪花算法） | 100001 |
| `parent_id` | 父主键（顶级为 0，子级指向父级 id） | 0 / 100001 |
| `code` | 字典编码（模块唯一标识） | 'message_type' |
| `dict_key` | 字典键值（**顶级固定为 -1**，子级为业务值） | -1 / 1 / 2 / 3 |
| `dict_value` | 字典显示名称 | '消息类型' / '通知' |
| `sort` | 排序 | 1, 2, 3... |
| `is_sealed` | 是否封存（0: 否，1: 是） | 0 |
| `status` | 状态（默认 1） | 1 |
| `is_deleted` | 是否删除（0: 否，1: 是） | 0 |

**参考现有字典数据**:
- 性别字典：`sex`，父级 `dict_key = '-1'`，子级 `dict_key = '1'` (男), `'2'` (女)

### 3.4 权限配置 SQL 脚本

**文件**: `sql/03_menu/menu_inner_message.sql`

> **注意**: BladeX 菜单表结构说明
> - 无自增 ID，使用**雪花算法生成** bigint 类型 ID
> - `parent_id`: 父级菜单 ID（顶级菜单为 0，子菜单指向父级 ID）
> - `category`: 菜单类型（1=一级菜单，2=二级/按钮菜单）
> - `action`: 操作按钮类型（1=工具栏，2=操作栏，3=工具操作栏）
> - 无 `create_time`、`menu_name`、`menu_code`、`menu_uri`、`menu_type`、`menu_icon`、`component_path`、`is_inner`、`is_datascope`、`permission` 字段

**菜单层级关系**:
```
顶级菜单 (parent_id = 0)
  └── 一级菜单 (parent_id = 顶级菜单 ID)
      └── 按钮/功能菜单 (parent_id = 一级菜单 ID)
```

**所属模块查找**:
- 如果模块有父级（如 `desk`），需要查找对应父级 ID
- 参考现有数据：`desk` 工作台 ID = `1123598815738675201`

```sql
-- =============================================
-- 菜单：内部消息
-- 所属模块：desk（工作台）
-- 父级菜单 ID: 1123598815738675201 (desk)
-- 创建时间：2026-04-02
-- =============================================

        boolean temp = innerMessageService.deleteLogic(Func.toLongList(ids));
        return R.status(temp);
    }

}
```

**要点**:
- ✅ `@TenantDS`：启用多租户数据源
- ✅ `@PreAuth(menu = "innermessage")`：权限控制（**必须与阶段 2.4 定义的菜单码一致**）
- ✅ `@RequestMapping(AppConstant.APPLICATION_DESK_NAME + "/innermessage")`：统一前缀
- ✅ `@Tag` 和 `@Operation`：Swagger 文档注解
- ✅ `Condition.getQueryWrapper()`：自动构建查询条件
